# Trial: R73 Q8_0 Direct Dot

Date: 2026-06-16
Owner: RLLM
Status: success with prefill limitation
Folder: active

## Hypothesis

R72 accepted `q8_transformer_keep_io` for chat quality and memory, but it still
dequantized each Q8_0 chunk into a transient F32 buffer before matmul. Directly
accumulating Q8_0 blocks should remove that F32 scratch window and improve
decode throughput without changing model bytes or generated text.

## Scope

- Mode: exact-lowram
- Model/artifact: `models/SmolLM2-135M-Instruct-q8_transformer_keepio.rllm`
- Architecture: SmolLM2/Llama-compatible decoder
- Target device/profile: local macOS CPU, release binary
- Bottleneck tag: Q8 dequantization scratch | memory bandwidth | decode CPU

## Change

- Added direct Q8_0 accumulation kernels for:
  - `streaming_tile_linear_from_model`
  - `streaming_tile_linear_multiply_into_from_model`
  - `streaming_tile_linear_argmax_from_model`
- Q8_0 blocks now read one fp16 scale plus 32 signed i8 weights and dot directly
  against F32 activations.
- Q4_0 still uses the existing dequantized F32 path.

## Comparison To R72 Q8

R72 Q8 transformer keep-IO:

- decode: about 14.44-14.69 tok/s
- RSS: about 382-383 MB
- peak transient: 116785152 bytes

R73 Q8 direct-dot:

- decode: 15.17-17.88 tok/s
- RSS: about 379-380 MB
- peak transient: 113246208 bytes
- output text unchanged from R72 Q8 for this matrix

## Decision

Accept direct Q8_0 accumulation for exact-lowram Q8 artifacts.

This removes the full F32 dequant scratch from Q8 projection paths, lowers peak
transient memory by about 3.5 MB on SmolLM2, lowers RSS by about 2-4 MB, and
improves decode throughput. TTFT/prefill remains the next bottleneck, so the
next useful optimization is row-blocked or SIMD direct-dot Q8 for prefill and
decode, not more Q4 MLP work.
