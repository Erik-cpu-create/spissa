# Trial: R141 — REEFLOW-BF16-DOT exact-bf16 GEMV (bfdot)

Date: 2026-06-19
Owner: RLLM
Status: inconclusive
Folder: inconclusive

## Hypothesis

Replacing the LM-head/embedding exact-bf16 GEMV (bf16→f32 upcast + f32 FMA, 4 lanes)
with an ARM `bfdot` kernel (8 bf16/instr → f32 accumulate) keeps weights bit-exact
and computes at native bf16 precision ~2× faster, improving `--fast` decode for the
tied-bf16-embedding models (LLaMA/Gemma) with zero weight-quality loss (vs q8's
0.56% RMS).

## Scope

- Mode: fast-lowram (opt-in experimental kernel)
- REE kernel: REEFLOW-BF16-DOT (working name; Erik's final call before any paper use)
- Model/artifact: `Llama-3.2-1B-Instruct-q8_transformer_keepio-rowchunks.spsa`
- Architecture: LLaMA 3.2 1B, tied bf16 embedding / LM head (hidden 2048, vocab 128256)
- Target device/profile: Apple A18 Pro (2 P + 4 E), macOS; FEAT_BF16 present
- Expected bottleneck: CPU bf16 GEMV arithmetic (f32-FMA → bfdot)
- Bottleneck tag: memory bandwidth (the decisive one, found by measurement)

## Analysis

The isolated kernel works and is real: bfdot is correct (bit-exact weights, rel err
< 1e-2 vs the f32 reference) and ~1.2× faster than the f32-upcast row-dot at
hidden=2048.

But end-to-end decode does not move. The decode LM head reads the full bf16 vocab
each token: 128256 × 2048 × 2 B ≈ 525 MB/token, taking ~22 ms/token ≈ **24 GB/s** —
i.e. it is **memory-bandwidth-bound**, not compute-bound. bfdot reads the same 525 MB
(weights are bf16 in both paths), so it cannot speed a bandwidth-bound GEMV. The
isolated 1.2× came from the compute/upcast side, which is not the decode bottleneck.

Two consequences made bfdot OPT-IN (default off) rather than the default `--fast`
kernel:
1. No decode speedup today (bandwidth wall).
2. bfdot rounds activations to bf16, which is slightly less precise than the
   f32-upcast `--fast` path. Defaulting it on would trade precision for no speed.

This is the negative result the feasibility framing exists to catch — and it sharpens
the project thesis: the decode lever for tied-bf16 LM heads is **bytes read**, not
compute. That is exactly Phase 2 (compressed-resident, `rtc-dfloat-v1`): decode fewer
bytes per row, at which point the GEMV stops being bandwidth-bound and bfdot's compute
win surfaces. bfdot is retained, validated, and opt-in as the compute half of that
fused decode.

## Decision

inconclusive (kernel accepted + retained; decode-speedup hypothesis not supported e2e)

Reason: The bfdot kernel is correct, exact-weight, and ~1.2× faster in isolation, but
the bf16 LM-head GEMV is memory-bandwidth-bound, so it yields no end-to-end decode
gain and would slightly reduce precision if defaulted on. Kept as an opt-in
(`RLLM_BF16_DOT=1`), validated building block for Phase 2 compressed-resident decode.

Paper value:

- use as limitation (bandwidth wall on the bf16 LM head) AND positive evidence for the
  Phase 2 direction (compute is cheap; bytes-read is the bottleneck — motivates
  lossless compressed-resident decode).

## Next Experiment

Phase 2: make `rtc-dfloat-v1` decode fast (buffered/SIMD bit-reader + per-row framing),
then fuse decode→bfdot so the LM head reads ~11 bits/weight instead of 16 — turning the
bandwidth wall into a bandwidth win where bfdot's compute speed finally matters.
